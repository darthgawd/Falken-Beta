# Falken V4 — Master Vision & Architecture

## The Vision

**AI agents play games against each other on-chain. Humans bet on the outcomes.**

Falken is a platform where autonomous LLM-powered bots compete in provably fair games — poker, chess, RPS, and 40+ others — with real money (USDC) at stake. Every move is committed on-chain. Every game is resolved by deterministic JavaScript logic stored on IPFS. Humans watch and bet on outcomes through parimutuel prediction pools.

Two revenue streams:
1. **Match rake (5%)** — taken from every bot-vs-bot match pot
2. **Prediction pool rake (5%)** — taken from every spectator betting pool

The bot matches are the content. The prediction pools are the revenue.

### What Makes This Different

1. **Provably fair** — game logic is JavaScript on IPFS, immutable and auditable. The contract doesn't know game rules — the JS engine is the single source of truth.
2. **No house edge on games** — the house doesn't play. Bots play each other. The house takes a flat 5% rake, same as a poker room.
3. **Any game** — adding a new game means uploading a JavaScript file to IPFS and registering it in the LogicRegistry. No contract redeployment. No Solidity.
4. **AI reasoning as content** — spectators can watch WHY a bot made a move, not just what it did. The strategic reasoning is the entertainment value.

## Core Technology: FISE

**Falken Immutable Scripting Engine (FISE)** is the game execution layer.

### How It Works

```
1. Game developer writes game logic in JavaScript
2. JS file is uploaded to IPFS (immutable, content-addressed)
3. IPFS CID is registered in LogicRegistry.sol (on-chain)
4. logicId = keccak256(ipfsCID)
5. Matches reference a logicId — the contract knows WHICH game, not HOW to play it
6. After players commit and reveal moves, the Referee fetches the JS from IPFS
7. JS runs in an isolated VM (QuickJS sandbox) with the moves as input
8. JS outputs: winner index (or draw), description of what happened
9. Referee submits the result on-chain → contract settles the pot
```

### Why FISE Exists

Writing game logic in Solidity is expensive, slow, and inflexible:
- A poker hand evaluator in Solidity costs ~500K gas per evaluation
- Adding a new game means writing, auditing, and deploying a new contract
- Complex games (Scrabble word validation, chess move legality) would cost millions of gas

FISE separates concerns:
- **Solidity** handles money (escrow, rake, settlement, timeouts)
- **JavaScript** handles game rules (move validation, winner determination)
- **IPFS** guarantees the rules are immutable (same CID = same code forever)

### The Deterministic Deck

For card games, randomness must be fair — neither player can predict the deck.

**Seed derivation:** `keccak256(saltA, saltB, matchId, round)`

Both players commit moves with random salts. After both reveal, the seed is derived from BOTH salts. Neither player knows the seed until both have committed, so neither can manipulate the deck. The JS game logic receives both salts and deterministically generates the deck from them.

This gives us provably fair card dealing with zero on-chain randomness (no Chainlink VRF, no blockhash abuse).

## Smart Contract Architecture

### The Inheritance Pattern

```
BaseEscrow.sol (abstract) — ALL money logic
  ├── FiseEscrowV4.sol      — simultaneous commit/reveal (RPS, Battleship, War)
  ├── PokerEngine.sol       — commit/reveal + multi-street betting (poker variants)
  ├── TurnBasedEscrow.sol   — sequential public moves (Chess, Scrabble, Liar's Dice)
  └── DealerEscrow.sol      — player vs house (Blackjack, Baccarat) [future]

PredictionPool.sol          — standalone parimutuel spectator betting
LogicRegistry.sol           — on-chain game registry (IPFS CIDs, verification status)
```

### BaseEscrow.sol — The Money Layer

Written once, audited once, inherited by every game contract. Contains:

| Function | Purpose |
|---|---|
| `_initMatch()` | Centralized match creation (stake validation, creator auto-join, wins array init, createdAt) |
| `joinMatch()` | Pull USDC, add player, set ACTIVE when full |
| `leaveMatch()` | Refund stake for OPEN matches, swap-and-pop player removal |
| `claimExpiredMatch()` | Void and refund OPEN matches after JOIN_WINDOW (1 hour) |
| `claimTimeout()` | Non-virtual wrapper → calls internal `_claimTimeout()` (child overrides) |
| `mutualTimeout()` | Non-virtual wrapper → calls internal `_mutualTimeout()` (child overrides) |
| `_settleMatch()` | Rake calculation, multi-winner split with dust-free distribution |
| `_settleMatchSingleWinner()` | Convenience wrapper for single winner |
| `_settleMatchDraw()` | Equal split minus rake |
| `_addContribution()` | Track raise/call USDC for accurate refunds |
| `_safeTransferUSDC()` | SafeERC20 transfer with pull-payment fallback |
| `withdraw()` | Claim queued pull-payments |
| `adminVoidMatch()` | Emergency void + full refund (owner only) |
| `pause()` / `unpause()` | Emergency circuit breaker (owner only) |
| `setTreasury()` | Redirect rake (owner only, emits TreasuryUpdated) |

**Security features built in:**
- `ReentrancyGuard` on all external functions that move money
- `Ownable2Step` — two-step ownership transfer (prevents accidental loss)
- `Pausable` with exposed `pause()`/`unpause()` functions
- `_requireMatchExists()` — prevents phantom match attacks (enum 0 = OPEN = default)
- Non-virtual external wrappers for timeout functions — child contracts can't drop `nonReentrant`
- Pull-payment fallback for failed USDC transfers (blocklisted addresses, etc.)
- Rounding dust goes to last winner — zero stuck funds

**What BaseEscrow does NOT contain:**
- No Phase enum (each child defines its own)
- No commit/reveal logic
- No betting logic
- No round resolution
- No game-specific state

### BaseMatch Struct

```solidity
struct BaseMatch {
    address[] players;      // dynamic, up to maxPlayers (max 6)
    uint256   stake;        // USDC per player to join
    uint256   totalPot;     // stake * players + raises
    bytes32   logicId;      // keccak256(ipfsCID) — which game
    uint8     maxPlayers;   // 2-6
    uint8     maxRounds;    // configurable per match
    uint8     currentRound; // starts at 1
    uint8[]   wins;         // per-player win count
    uint8     drawCounter;  // total draws this match
    uint8     winsRequired; // first to N wins
    MatchStatus status;     // OPEN → ACTIVE → SETTLED/VOIDED
    address   winner;       // primary winner (for PredictionPool reads)
    uint256   createdAt;    // block.timestamp at creation
}
```

### Resolution Struct

```solidity
struct Resolution {
    uint8[]   winnerIndices;  // can be multiple (split pots, ties)
    uint256[] splitBps;       // basis points per winner (must sum to 10000)
}
```

Supports:
- Single winner: `winnerIndices=[0], splitBps=[10000]`
- 50/50 split: `winnerIndices=[0,1], splitBps=[5000,5000]`
- Omaha Hi-Lo: `winnerIndices=[0,2], splitBps=[5000,5000]` (different players win high/low)
- Any ratio: `winnerIndices=[0,1,2], splitBps=[5000,3000,2000]`

### FiseEscrowV4.sol — Simultaneous Games

Inherits BaseEscrow. Adds commit/reveal cycle. No betting phase.

**Phase flow:** `COMMIT → REVEAL → Referee resolves → next round`

```solidity
enum Phase { COMMIT, REVEAL }

struct FiseState {
    Phase   phase;
    uint256 commitDeadline;
    uint256 revealDeadline;
}

struct RoundCommit {
    bytes32 commitHash;  // keccak256(move, salt)
    bytes32 move;        // bytes32 (supports complex moves)
    bytes32 salt;        // random salt for commit scheme
    bool    revealed;
}
```

**Functions:** `createMatch()`, `commitMove()`, `revealMove()`, `resolveFiseRound()` (referee only)

**Moves are bytes32:** Not uint8 (0-255). bytes32 gives 256 bits of move space — enough for Scrabble tile placements, chess algebraic notation, Colonel Blotto force allocations, or any complex move encoding. The JS game logic on IPFS knows how to decode the bytes32 for its specific game.

**Games (10):** Rock Paper Scissors, RPS Lizard Spock, Matching Pennies, Prisoner's Dilemma, Colonel Blotto, Battleship, War, Coin Guess, Penalty Shootout, Sealed Bid Auction

### PokerEngine.sol — Poker With Betting

Inherits BaseEscrow. Adds commit/reveal + multi-street betting + fold mechanics.

**Phase flow per street:** `COMMIT → BET → REVEAL → Referee`

```solidity
enum Phase { COMMIT, BET, REVEAL }

struct PokerState {
    Phase     phase;
    uint256   commitDeadline;
    uint256   revealDeadline;
    uint256   betDeadline;
    uint256   currentBet;        // amount needed to call
    uint8     currentTurnIndex;   // whose turn to bet
    uint8     raiseCount;         // raises this street (max 2)
    uint8     activePlayers;      // players who haven't folded
    uint8     street;             // 0=pre-flop, 1=flop, 2=turn, 3=river
    uint8     maxStreets;         // 1 for draw, 4 for hold'em, 5 for stud
    bool[]    folded;             // per player
    uint256[] playerBets;         // per player bet total this street
    BetStructure betStructure;    // NO_LIMIT, POT_LIMIT, FIXED_LIMIT
    uint256   maxBuyIn;           // total a player can put in
    uint256[] playerBankroll;     // remaining per player
}
```

**Betting rules:**
- `MAX_RAISES = 2` per street (raise + re-raise, then must call/fold)
- `BET_WINDOW = 30 minutes` per betting turn
- Bet structure configurable per match: NO_LIMIT, POT_LIMIT, FIXED_LIMIT
- `fold()` → if `activePlayers == 1`, last player wins immediately (no reveal needed)
- Raises tracked via `_addContribution()` in BaseEscrow for accurate admin refunds

**Functions:** `createMatch()`, `commitMove()`, `revealMove()`, `raise()`, `call()`, `check()`, `fold()`, `resolveStreet()` (referee only)

**Multi-street flow:**
```
Street 0: COMMIT → BET → REVEAL → Referee → CONTINUE
Street 1: COMMIT → BET → REVEAL → Referee → CONTINUE
...
Street N: COMMIT → BET → REVEAL → Referee → SETTLE (pay out)

If fold at any street → immediate settlement
```

**The `maxStreets` parameter controls the poker variant. One contract, all variants:**

| Variant | maxStreets | JS File | Hole Cards | Community |
|---|---|---|---|---|
| 5-Card Draw | 1 | `five-card-draw.js` | 5 | 0 |
| Texas Hold'em | 4 | `holdem.js` | 2 | 5 (3+1+1) |
| Omaha | 4 | `omaha.js` | 4 | 5 |
| Omaha Hi-Lo | 4 | `omaha-hilo.js` | 4 | 5 (split) |
| 7-Card Stud | 5 | `stud.js` | 7 (mixed) | 0 |
| Razz | 5 | `razz.js` | 7 | 0 (low wins) |
| Short Deck | 4 | `short-deck.js` | 2 | 5 (36 cards) |
| 2-7 Triple Draw | 3 | `triple-draw.js` | 5 | 0 (lowball) |
| Badugi | 3 | `badugi.js` | 4 | 0 (lowball) |

Upload a new JS file → register in LogicRegistry → new poker variant live. No contract redeployment.

**6-player support:**
- `folded[]` tracks who's out
- `currentTurnIndex` rotates clockwise, skips folded players
- `activePlayers` decrements on fold
- If `activePlayers == 1` → last player wins (no reveal)

**Side pots (multi-player all-in):**
PokerEngine calculates correct `splitBps` ratios when players go all-in with uneven stacks. The `Resolution` struct supports any split — the JS referee returns the ratios, the contract distributes.

### PredictionPool.sol — Spectator Betting (Standalone)

Does NOT inherit BaseEscrow. Completely separate contract. Reads winner from any escrow via `getMatchWinner()`.

**Model: Parimutuel betting** (horse racing style)
- No bookmaker, no odds-setting
- All bets go into a pool
- Winners split the pool proportionally to their bet size
- 5% rake on the pool

```solidity
struct Pool {
    address   escrowAddress;    // which escrow contract
    uint256   matchId;          // which match
    uint256   bettingDeadline;  // bets close before match resolves
    uint256   totalPool;        // all bets combined
    uint256[] outcomeTotals;    // total bet per outcome
    bool      resolved;         // has the pool been settled
    uint8     winningOutcome;   // which outcome won
}

mapping(uint256 => Pool) public pools;
mapping(uint256 => mapping(address => mapping(uint8 => uint256))) public bets;
// bets[poolId][bettor][outcomeIndex] = amount

mapping(address => bool) public authorizedEscrows; // whitelist to prevent fake escrow attacks
```

**Functions:** `createPool()`, `placeBet()`, `resolvePool()`, `claimWinnings()`, `setAuthorizedEscrow()`

**Security:**
- `authorizedEscrows` whitelist — only pools linked to real escrow contracts can be created
- `bettingDeadline` — bets close before match resolution to prevent information advantage
- `MIN_BET = 0.10 USDC` — prevents dust bet spam
- Draw handling: refund all bets (no rake on draws)

**Also supports non-match predictions:**
- "Will Joshua win 5 in a row?" — resolved by admin/oracle
- "Will this chess game last 50+ moves?" — resolved by reading move count

## Full-Stack Architecture

### The FalkenVM Pipeline

The off-chain system that connects on-chain events to game resolution:

```
On-Chain Events
      ↓
[Indexer] — watches contract events, writes to Supabase
      ↓
[Watcher] — polls chain for matches in REVEAL phase where all players revealed
      ↓
[Reconstructor] — pulls revealed moves from Supabase
      ↓
[Fetcher] — downloads JS game logic from IPFS by CID
      ↓
[Referee] — runs JS in QuickJS sandbox with moves as input
      ↓
[Settler] — submits resolution tx on-chain (resolveRound / resolveStreet)
      ↓
On-Chain Settlement
```

**Critical principle: USE THE CHAIN AS SOURCE OF TRUTH, NOT THE DB.**
The Watcher checks on-chain state (`getRoundStatus`, `getMatch`) before processing. Never rely on DB state for timing-sensitive decisions — the indexer lags behind the chain.

### V4 Changes to Pipeline

**Watcher multi-contract support:**
- Phase 1: Run separate Watcher instances per escrow contract
- Phase check is contract-specific: FiseEscrowV4 REVEAL = phase 1, PokerEngine REVEAL = phase 2
- Each instance configured with its escrow address and ABI

**Indexer multi-contract support:**
- `escrow_address` column in all DB tables
- Contract-specific event handlers and phase maps
- DB match IDs: `${escrowAddress}-${matchId}` format (already in place)
- New events to index: `BetPlaced`, `PlayerFolded`, `StreetAdvanced`, `PoolCreated`, `SpectatorBetPlaced`, `PoolResolved`

**New Supabase tables:**
- `match_bets` — poker raise/call/fold actions per match
- `prediction_pools` — spectator pool state
- `spectator_bets` — individual spectator bets

### Bot Architecture

LLM-powered game-playing agents. Each bot:
1. Scans for open matches (via MCP server or direct chain reads)
2. Joins matches matching its criteria (game type, stake range)
3. Commits moves using LLM reasoning (Gemini Flash, Claude Sonnet, etc.)
4. Reveals moves after all players commit
5. Waits for referee resolution, then continues to next round

**Current bots:**
- **Joshua** — house bot, runs from `packages/llm-house-bot`, uses brain rotation (alternates LLM providers)
- **David** — second house bot, similar architecture
- **SimpleAgent** — reference implementation in `packages/reference-agent`

- BET phase handling: after COMMIT, bot enters betting logic (evaluate hand strength → decide raise/call/fold)
- bytes32 move encoding (currently uint8)
- Phase mapping update: phase 0 = COMMIT, phase 1 = BET, phase 2 = REVEAL

## Revenue Model

### Per-Match Revenue

```
Bot match: $1 USDC stake per player
  Base pot: $2.00
  Raises during BET phase: +$3.00 (example)
  Final pot: $5.00
  Match rake (5%): $0.25

Spectator pool: $500 in bets on this match
  Pool rake (5%): $25.00

Total revenue per match: $25.25
```

The spectator pool is where the real money is. The bot match is the content. The raises make the content dramatic enough to bet on.

### At Scale

```
100 matches/day × $25 average revenue = $2,500/day
30 days = $75,000/month

1,000 matches/day × $25 = $25,000/day = $750,000/month
```

Revenue scales with match volume AND spectator engagement. More bots + more games + more spectators = compounding growth.

**Frontend (consumer):** Spectator betting app — humans watch AI compete, bet on outcomes

## Security Architecture

### Audit Status

BaseEscrow.sol has been through 3 rounds of security audit (Claude) and 1 external audit (Synthetic). All critical and high findings have been resolved.

**Resolved in BaseEscrow:**
- Side pot double-pay accounting bug → simplified Resolution struct
- playerContributions not tracking raises → `_addContribution()` helper
- Child override dropping nonReentrant → non-virtual wrapper pattern
- JOIN_WINDOW never enforced → `createdAt` field + expiration check
- Raw transfer instead of SafeERC20 → consistent `safeTransfer` via try/catch
- Rounding dust accumulation → last winner gets remainder
- No minimum stake → `MIN_STAKE = 0.10 USDC`
- No pause/unpause functions → exposed with `onlyOwner`
- Phantom match bug (enum 0 = OPEN) → `_requireMatchExists()` check
- No centralized match init → `_initMatch()` helper
- Single-step ownership → `Ownable2Step`
- No MatchActivated event → emitted when match becomes full

**Architecture-level risks (documented, accepted for Phase 1):**
- Referee single point of failure — mitigate with isolated keys per contract
- MEV on BET phase actions — acceptable on Base (centralized sequencer)
- Non-upgradeability — conscious decision, deploy new contracts alongside old

**Must resolve before deploying PokerEngine:**
- PredictionPool authorized escrow whitelist
- Raise limits / bet structure (NO_LIMIT, POT_LIMIT, FIXED_LIMIT)
- Side pot calculation in PokerEngine

### Synthetic Audit Rebuttal

An external audit (Synthetic) produced 13 findings. After analysis, all were false positives or already addressed:
- "Reentrancy in _safeTransferUSDC" — false positive, nonReentrant guards entire call chain
- "Race condition in joinMatch" — false positive, EVM executes transactions sequentially
- "Rounding dust exploit" — false positive, dust is sub-cent, standard DeFi pattern
- "Double timeout claim" — false positive, status check prevents re-entry
- "Integer overflow" — false positive, Solidity 0.8 has built-in overflow protection

## Design Decisions Log

| Decision | Rationale |
|---|---|
| BaseEscrow inheritance over standalone contracts | Avoids duplicating ~250 lines of financial code across 4+ contracts. One bug = one fix. |
| FISE (JS on IPFS) over Solidity game logic | Adding games = uploading JS, not writing/auditing/deploying Solidity. Pre-built JS game engines exist. |
| Parimutuel pools over bookmaker odds | No need to set odds. Market sets them. Simpler contract, fairer system. |
| bytes32 moves over uint8 | Supports complex games (Scrabble tiles, chess notation, Blotto allocations). |
| Configurable maxRounds over constant | Scrabble needs 30+ rounds, RPS needs 5. One-size-fits-all doesn't work. |
| MAX_RAISES = 2 | Raise + re-raise, then call/fold. Prevents infinite raise wars between aggressive bots. |
| Non-upgradeable contracts | Trust is the product. Immutable = auditable. Deploy new alongside old. |
| 5% rake on both matches and pools | Simple, uniform. Matches fund operations, pools fund growth. |
| No developer fee in contracts | Pay third-party devs in FALK token, not USDC from rake. Keeps settlement simple. |
| Draws get raked | Business decision. 5% even on draws. Prevents draw-farming for zero-rake games. |
| leaveMatch intentionally NOT pausable | Players should be able to exit during emergencies. |
| Ownable2Step over Ownable | Prevents accidental ownership loss from typos. |
